A Arte de Especificar

Objetivo

Dominar a arte de escrever prompts eficazes para sistemas agenticos. Transformar instrucoes vagas em especificacoes precisas que o agente executa com consistencia.

"If you get the spec right, the code almost writes itself."

Al Harris, Amazon Kiro (AI Engineer Summit 2024)

Lema da Aula

"Se voce acertar a especificacao, o codigo quase escreve a si mesmo."

Pre-requisitos

  • Modulo 1 completado (Aulas 0-5)
  • Experiencia basica com Claude Code
  • Compreensao de CLAUDE.md e contexto

Bloco 1: Paradigma Spec-Driven

25 min

1.1 Vibe Coding vs Vibe Engineering

"Vibe Coding e quando voce nao sabe o que esta fazendo e deixa a IA fazer tudo. Vibe Engineering e quando voce sabe exatamente o que quer e usa IA para acelerar."

Kitze (AI Engineer Summit 2025) 
┌─────────────────────────────────────────────────────────────────┐
│              VIBE CODING vs VIBE ENGINEERING                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   VIBE CODING (Anti-padrao):                                    │
│   ─────────────────────────────                                 │
│   "Gere uma minuta de sentenca para este processo de BPC"       │
│                                                                 │
│   O que acontece:                                               │
│   • Agente ADIVINHA o formato                                   │
│   • Agente INVENTA estrutura                                    │
│   • Agente OMITE informacoes essenciais                         │
│   • Voce RETRABALHA 80% do output                               │
│                                                                 │
│   VIBE ENGINEERING (Padrao correto):                            │
│   ─────────────────────────────────────                         │
│   "Gere minuta de sentenca com:                                 │
│    - Estrutura: RELATORIO → FUNDAMENTACAO → DISPOSITIVO         │
│    - Requerente: [nome completo]                                │
│    - Fundamento: Art. 20 da Lei 8.742/93                        │
│    - Citar: Tema 1038 do STF                                    │
│    - Formato: Paragrafos numerados, sem bullets                 │
│    SEMPRE verificar se ha laudo social nos autos."              │
│                                                                 │
│   O que acontece:                                               │
│   • Agente SEGUE o formato                                      │
│   • Agente USA estrutura definida                               │
│   • Agente INCLUI informacoes especificadas                     │
│   • Voce REVISA 20% (ajustes finos)                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

1.2 O Framework Spec-Driven

Specs sao o contrato entre voce e o agente. Quanto mais claro o contrato, melhor a entrega.

┌─────────────────────────────────────────────────────────────────┐
│                  FRAMEWORK SPEC-DRIVEN                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   FASE 1: REQUIREMENTS (O QUE)                                  │
│   ──────────────────────────────                                │
│   • User Story: "Como [papel], quero [acao] para [beneficio]"   │
│   • Criterios de Aceitacao: Lista de verificacoes               │
│   • Restricoes: O que NAO deve acontecer                        │
│                                                                 │
│   FASE 2: DESIGN (COMO)                                         │
│   ─────────────────────                                         │
│   • Entrada esperada: Formato, tamanho, tipo                    │
│   • Saida esperada: Template, estrutura, campos                 │
│   • Fluxo: Passos em ordem                                      │
│                                                                 │
│   FASE 3: TASKS (PASSOS)                                        │
│   ─────────────────────                                         │
│   • Cada tarefa: Uma acao verificavel                           │
│   • Dependencias: O que precisa vir antes                       │
│   • Criterio de sucesso: Como saber que terminou                │
│                                                                 │
│   ┌─────────────┐     ┌─────────────┐     ┌─────────────┐       │
│   │REQUIREMENTS │ ──▶ │   DESIGN    │ ──▶ │   TASKS     │       │
│   │  (O que)    │     │  (Como)     │     │  (Passos)   │       │
│   └─────────────┘     └─────────────┘     └─────────────┘       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
Aspecto Vibe Coding Vibe Engineering
Voce sabe O que quer vagamente Exatamente o que quer
Voce especifica Resultado final Resultado + estrutura + regras
Agente faz Adivinhacao criativa Execucao precisa
Retrabalho 80% (frustrante) 20% (refinamento)

2.2 Exemplo: Prompt Fraco vs Forte

Prompt FRACO

"Analise este documento e me de um resumo"

Problemas: Nao define formato, nao especifica o que incluir, nao estabelece restricoes.

Prompt FORTE

"Analise este documento e gere um resumo estruturado com:
1. FATOS: Lista cronologica dos eventos
2. PARTES: Nome e qualificacao de cada parte
3. PEDIDOS: Itemizacao do que e solicitado
4. FUNDAMENTOS: Base legal citada

SEMPRE incluir pagina de referencia.
NUNCA omitir valores monetarios.
PREFERIR citacoes diretas a parafrases."

Bloco 3: Regras Explicitas

25 min

3.1 O Padrao SEMPRE/NUNCA/PREFERIR

"As regras sao como instrucoes para um novo estagiario. Quanto mais explicitas, menos erros. O que e 'obvio' para voce nao e 'obvio' para o agente."

Barry Zhang, Anthropic (AI Engineer Summit) 
┌─────────────────────────────────────────────────────────────────┐
│              PADRAO SEMPRE/NUNCA/PREFERIR                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   SEMPRE (Obrigatorio - nunca pular)                            │
│   ─────────────────────────────────────                         │
│   • SEMPRE incluir referencia a pagina dos autos                │
│   • SEMPRE citar o fundamento legal completo                    │
│   • SEMPRE verificar se existe laudo pericial                   │
│                                                                 │
│   NUNCA (Proibido - erro grave se fizer)                        │
│   ─────────────────────────────────────                         │
│   • NUNCA omitir valores monetarios                             │
│   • NUNCA inventar dados nao presentes nos autos                │
│   • NUNCA incluir opiniao pessoal                               │
│                                                                 │
│   PREFERIR (Prioridade - quando houver escolha)                 │
│   ────────────────────────────────────────────                  │
│   • PREFERIR citacao direta a parafraseamento                   │
│   • PREFERIR ordem cronologica a tematica                       │
│   • PREFERIR bullet points a paragrafos longos                  │
│                                                                 │
│   Por que funciona:                                             │
│   ─────────────────                                             │
│   • LLMs processam instrucoes explicitas melhor que implicitas  │
│   • SEMPRE/NUNCA criam restricoes fortes (hard constraints)     │
│   • PREFERIR cria heuristicas de desempate (soft constraints)   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

3.2 Cobertura de Casos

Tipo de Regra O que cobre Exemplo
Requisitos O que TEM que estar presente SEMPRE citar artigo de lei
Restricoes O que NAO pode acontecer NUNCA inventar jurisprudencia
Prioridades Desempate entre opcoes PREFERIR STJ sobre TRF

4.2 Feedback Fraco vs Feedback Forte

Feedback Fraco Feedback Forte
"Esta errado" "O campo DATA esta no formato DD/MM/YYYY mas deveria ser YYYY-MM-DD"
"Faltou informacao" "Faltou o campo VALOR_CAUSA que aparece na linha 47 da peticao inicial"
"Ficou confuso" "A secao FATOS misturou alegacoes do autor com do reu - separar em FATOS_AUTOR e FATOS_REU"
Regra de Ouro

Se voce nao consegue explicar O QUE esta errado e COMO deveria ser, o feedback nao vai melhorar o prompt.

Bloco 5: Os 10 Erros Mais Comuns

20 min

Lista de Anti-Padroes

┌─────────────────────────────────────────────────────────────────┐
│              OS 10 ERROS MAIS COMUNS                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. VAGUEZA EXCESSIVA                                           │
│     ✗ "Analise o documento"                                     │
│     ✓ "Extraia os 5 pedidos principais do documento"            │
│                                                                 │
│  2. AUSENCIA DE FORMATO                                         │
│     ✗ "Me de um resumo"                                         │
│     ✓ "Gere resumo em formato: ## SECAO > ### SUBSECAO"         │
│                                                                 │
│  3. REGRAS IMPLICITAS                                           │
│     ✗ (assumir que agente sabe o "obvio")                       │
│     ✓ "SEMPRE incluir referencia de pagina"                     │
│                                                                 │
│  4. CONTEXTO INSUFICIENTE                                       │
│     ✗ "Gere uma minuta"                                         │
│     ✓ "Gere minuta de sentenca de BPC para idoso urbano"        │
│                                                                 │
│  5. FALTA DE EXEMPLOS                                           │
│     ✗ "Use o formato correto"                                   │
│     ✓ "Use este formato: [EXEMPLO CONCRETO]"                    │
│                                                                 │
│  6. INSTRUCOES CONTRADITORIAS                                   │
│     ✗ "Seja conciso mas inclua todos os detalhes"               │
│     ✓ "Maximo 500 palavras, priorizando: X, Y, Z"               │
│                                                                 │
│  7. FEEDBACK VAGO                                               │
│     ✗ "Esta errado, refaca"                                     │
│     ✓ "O campo X esta incorreto. Deveria ser Y porque Z"        │
│                                                                 │
│  8. NAO ITERAR                                                  │
│     ✗ Esperar resultado perfeito na primeira tentativa          │
│     ✓ Planejar 5-10 iteracoes de refinamento                    │
│                                                                 │
│  9. IGNORAR EDGE CASES                                          │
│     ✗ So testar com casos "normais"                             │
│     ✓ Testar com: caso vazio, caso gigante, caso estranho       │
│                                                                 │
│ 10. NAO DOCUMENTAR                                              │
│     ✗ Manter prompts so na memoria                              │
│     ✓ Salvar em Commands/Skills com versionamento               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

Checklist de Validacao

Antes de considerar o prompt pronto
  • Tem CONTEXTO explicito?
  • Tem TAREFA clara com verbo de acao?
  • Tem REGRAS no padrao SEMPRE/NUNCA/PREFERIR?
  • Tem FORMATO de saida definido?
  • Tem pelo menos 1 EXEMPLO concreto?
  • Passou pelo Learning Loop (5+ iteracoes)?
  • Foi testado com edge cases?
  • Esta documentado em Command ou Skill?

Resumo da Aula

Conceitos-Chave

  • Spec-Driven: Especificacao clara antes da execucao
  • Vibe Engineering: Saber exatamente o que quer
  • SEMPRE/NUNCA/PREFERIR: Padrao para regras explicitas
  • Prompt Learning Loop: Escrever → Rodar → Avaliar → Refinar
  • Convergencia: 5 execucoes consecutivas com output aceitavel
